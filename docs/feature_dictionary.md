# Feature Dictionary

Every feature in `data/processed/features.parquet`, generated by
`backend/app/features/build_features.py` from the raw 24-column dataset.
**All features use only the provided columns ‚ÄĒ no external data of any kind**
(see DECISIONS.md ADR-001). Statistics below are computed on the real
298,445-row dataset (5 rows dropped for missing `created_datetime`; see
`docs/data_quality_report.md`).

Targets (`data/processed/targets.parquet`) are documented separately at the
bottom ‚ÄĒ they intentionally use future data and must never be used as model
features.

## Spatial features (`features/spatial.py`)

| Feature | Formula | Source columns | Leakage risk | Expected impact | Internal-only? |
|---|---|---|---|---|---|
| `h3_cell` | H3 hexagonal cell ID, resolution 9 (~174m edge) | `latitude`, `longitude` | None | Primary spatial key for every grouped feature below and for Phase 3 modeling | ‚úÖ |
| `geohash` | GeoHash, precision 7 (~153m √ó 153m) | `latitude`, `longitude` | None | Comparison index, not primary model key (ADR-002) | ‚úÖ |
| `hotspot_frequency` | Count of prior violations in this `h3_cell`, strictly before this row's timestamp | `h3_cell`, `created_datetime` | **Leakage-safe** (expanding, excludes self) ‚ÄĒ see ADR-006 | Strongest spatial-risk signal; ranges 0‚Äď12,122, median 363 | ‚úÖ |
| `violation_density` | `hotspot_frequency / days_observed_so_far_in_cell` | same as above | Leakage-safe | Normalizes frequency by how long we've been watching that cell | ‚úÖ |
| `junction_density` | Cumulative count of *distinct* `junction_name` values seen in this cell, before this row | `h3_cell`, `junction_name`, `created_datetime` | Leakage-safe | Structural diversity of enforcement context per cell | ‚úÖ |
| `police_station_density` | Same pattern, distinct `police_station` values | `h3_cell`, `police_station`, `created_datetime` | Leakage-safe | Median 1, max 4 ‚ÄĒ most cells map to a single station | ‚úÖ |

2,534 distinct H3 cells / 5,753 distinct GeoHash cells found in the dataset
(GeoHash's finer grid splits some H3 cells further ‚ÄĒ expected given the
different cell shapes, ADR-002).

## Rolling features (`features/rolling.py`)

| Feature | Formula | Source columns | Leakage risk | Expected impact | Internal-only? |
|---|---|---|---|---|---|
| `violations_last_15m` / `_30m` / `_60m` | Count of OTHER violations in same `h3_cell` within the preceding N minutes (`rolling(window, closed='left')`) | `h3_cell`, `created_datetime` | **Leakage-safe** ‚ÄĒ verified by brute-force unit test (`tests/test_features.py`) | Direct short-term congestion-building signal for Phase 4 forecast windows | ‚úÖ |
| `same_hour_previous_day` | Count of violations in same `h3_cell`, same hour-of-day, exactly 1 calendar day earlier | `h3_cell`, `created_datetime`, `hour` | Leakage-safe (always looks backward 1 day) | Captures daily seasonality per cell/hour | ‚úÖ |
| `rolling_hotspot_intensity` | Hawkes-process-style decayed sum: each past event contributes 1.0, decaying with a 24h halflife; current row excluded | `h3_cell`, `created_datetime` | **Leakage-safe** ‚ÄĒ see rolling.py docstring for why a naive `ewm().mean()` was wrong and replaced | Smooth, continuously-decaying "how hot is this cell right now" signal, more reactive than a fixed count to recent bursts | ‚úÖ |

## ‚ö†ÔłŹ Targets (`data/processed/targets.parquet`) ‚ÄĒ NOT features

These columns use **future** data on purpose (that's the definition of a
supervised label) and must never be joined into a training feature matrix.
See DECISIONS.md ADR-005 for the full rationale.

| Column | Formula | Leakage risk | Use |
|---|---|---|---|
| `target_count_15m` / `_30m` / `_60m` | Count of OTHER violations in the same `h3_cell` in the N minutes AFTER this row's timestamp | **Uses future data ‚ÄĒ TARGET ONLY** | Phase 4 forecast regression target |
| `target_hotspot_60m` | `1` if `target_count_60m > 0` else `0` | **Uses future data ‚ÄĒ TARGET ONLY** | Phase 3 hotspot classification target |
